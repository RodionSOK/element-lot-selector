# Element — Выборщик лотов

Сервис с приёмом броней и заявок. Также есть админка для управления лотами и обращениями. Бэкенд — Python/FastAPI (REST + JSON-RPC), фронтенд — React 19 + TypeScript + Vite + Tailwind, БД — PostgreSQL.

## Запуск

Понадобится Docker и Docker Compose.

```bash
cp .env.example .env
```

Откройте `.env` и задайте свои значения секретов (`POSTGRES_PASSWORD`, `JWT_SECRET`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`) — технические параметры (хост/порт БД, CORS и т.д.) можно оставить по умолчанию.

```bash
docker compose up --build
```

Поднимутся три контейнера: `db` (Postgres), `backend` (FastAPI на `:8000`, миграции применяются автоматически при старте), `frontend` (статическая сборка на nginx, `:5173`).

Чтобы добавить лоты нужно:

1. Открыть `http://localhost:5173/admin/login`, войти под `ADMIN_USERNAME`/`ADMIN_PASSWORD` из `.env`.
2. На вкладке «Фиды» загрузить XML — файлом или по ссылке (например, `https://element.allio.agency/feeds/element.xml`).
3. Активировать созданный набор — после этого лоты появятся в публичном каталоге (`http://localhost:5173/`).

## Отчёт

### Задача, границы, допущения, план

**Задача.** Сквозной срез платформы: БД, API, UI, с двумя ролями — публичный посетитель (каталог, бронь, заявка) и админ (загрузка фида, активация набора, просмотр обращений).

**Границы.** 
- Использовалась стандартная палитра Tailwind. Дизайн не разрабатывался.
- Реализована только авторизация админа, регистрации нет.
- Админ может фильтровать заявки и брони по статусу, управление заявками и бронями не реализовано.

**Допущения:**
- Реальный фид (`element.xml`, 3685 лотов): Часть `RESERVED`-лотов с ценой 0, несколько записей с нереалистичной площадью (1000+ м²). Такие записи не фильтруются и не отбрасываются — принимаются как есть; записи с отсутствующими обязательными полями (например, без `flat_id`) пропускаются при импорте с логированием, а не роняют весь импорт.
- Цена за м² не хранится, вычисляется в SQL на лету (`price / NULLIF(area, 0)`, с защитой от деления на ноль).
- Для заявки и брони используется одна и та же схема (конкретно в коде - общий класс ContactInfo).

**План.** 
- Разобрана структура `element.xml` (`object → building → flat`, статусы `FREE/RESERVED/SOLD`)
- Таблицы (`lot_sets`, `lots`, `applications`, `bookings`) и конфигурация. 
- Бэкенд слоями (config → db → models → alembic → schemas → services → REST/RPC-адаптеры → тесты).
- Фронтенд 
- Docker-упаковка.

### REST vs JSON-RPC

Оба транспорта реализуют одну и ту же бизнес-логику через общий сервис-слой, но закладывают разные модели взаимодействия:

| | REST | JSON-RPC |
|---|---|---|
| Адресация | ресурс + HTTP-метод (`POST /lots/{id}/bookings`) | один endpoint (`/rpc`), метод — строка в теле (`bookings.create`) |
| Коды ответа | настоящие HTTP-статусы (404/409/401) | всегда `200`, ошибка — внутри JSON-тела со своим кодом |
| Файлы | нативный `multipart/form-data`, `UploadFile` | нет способа передать бинарные данные напрямую — фид кодируется в base64 и кладётся в JSON |
| Расширение | новый роут + новая функция | новый метод в реестре (`@rpc_method`) — диспетчер не меняется |

**Где легче:** REST удобнее и проще, если нужна привычная HTTP-семантика (статус-коды, готовый тулинг), и где есть бинарные данные (загрузка фида файлом).

**Где сложнее:** JSON-RPC: нет готовых кодов ошибок, поэтому были использованы придуманные коды для базовых ситуаций. Также дублируется обработка ошибок. Для фронтенда создается табличка-переводчик, чтобы браузер смог обрабатывать ошибки. При отправке файла его надо переводить в текстовую строку (base64), так как JSON-RPC может пересылать только текст.

### Конфигурация

Секреты (`POSTGRES_PASSWORD`, `JWT_SECRET`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`) и технические параметры (`POSTGRES_HOST`, `POSTGRES_PORT`, `LOG_LEVEL`, `CORS_ORIGINS`) лежат в одном `.env`. У техпараметров есть безопасные дефолты в коде (`Settings` в `config.py`), у секретов дефолтов нет намеренно.

**ни одно составное значение не хранится отдельной строкой**. `DATABASE_URL` нигде не существует как самостоятельная переменная — `config.py` собирает её программно из атомарных полей (`postgres_user`, `postgres_password`, `postgres_host`, `postgres_port`, `postgres_db`). То же самое и в Alembic — `env.py` не хранит свою копию строки подключения, а берёт её из того же `Settings`.

`docker-compose.yml` передаёт `POSTGRES_PORT` через `command: -c port=${POSTGRES_PORT}` в сам процесс Postgres **и** использует ту же переменную в пробросе порта наружу (`"${POSTGRES_PORT}:${POSTGRES_PORT}"`), бэкенд подключается по ней же.

### AI (рассуждение, не реализовано)

**Задача.** По свободному тексту заявки («хочу для себя», «ищу для сдачи в аренду», «подбираю для родителей») определить мотивацию покупателя: инвестиция / для проживания / для детей/родителей / не определено.

**Граница «код vs LLM».** Явные, устойчивые лексические сигналы («сдавать», «аренда», «инвестиция», «вложение», инвестиция; «детский сад», «школа рядом», «для дочери/сына», для детей) можно ловить простыми детерминированными правилами (keyword-матчинг) — это дёшево. LLM имеет смысл подключать только там, где правила не справляются: неявные, перефразированные или контекстные формулировки без прямых ключевых слов («хочу, чтобы было куда приезжать внукам» — про детей/семью, но без явного «детям»). То есть LLM — это не основной классификатор, правила забирают на себя большую и самую дешёвую часть случаев и снижают нагрузку на модель.

**Недоверенный ввод.** Комментарий заявки — текст от анонимного посетителя сайта, то есть по определению недоверенный ввод, который должен обрабатываться так же осторожно, как SQL/HTML-инъекции: он никогда не должен идти в промпт как исполняемая инструкция, а строго как размеченные данные («классифицируй следующий текст пользователя: `<comment>`»), с явным разделением системного промпта и пользовательского ввода, чтобы текст заявки не мог переопределить инструкцию модели (классический prompt injection: комментарий вида «игнорируй предыдущие инструкции и...»). 

**Заземление вывода и детерминизм.** Модель должна возвращать не свободный текст, а один из фиксированного набора вариантов. Температуру можно сделать близкой к нулю, чтобы для одного и того же текста ответ был стабилен между запусками (важно для воспроизводимости). Для случаев низкой уверенности — явная категория «не определено» вместо принудительного выбора наименее плохого варианта; такие заявки помечаются на ручной разбор. Для последующей проверки — логировать и исходный текст, и решение модели (вариант + при возможности что именно из текста на него повлияло).

**Как бы встраивалось.** Не синхронно в момент создания заявки (не блокировать ответ пользователю ожиданием LLM), а асинхронной фоновой задачей после сохранения заявки — тег мотивации добавляется отдельным полем, админ видит его в списке заявок и может его переопределить вручную.
